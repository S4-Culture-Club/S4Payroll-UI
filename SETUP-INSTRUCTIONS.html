Step-by-step setup

1 Create a new GitHub repo
Go to github.com/new
Name it s4labour-payroll (or anything you like)
Set it to Public
Do not initialise with README (you're bringing your own files)
2 Build the folder structure on your computer

Create these folders:
s4labour-payroll/
s4labour-payroll/bureau/
s4labour-payroll/org/
s4labour-payroll/shared/
s4labour-payroll/.github/workflows/

Then rename and move each downloaded file into place (see table above).
Also create an empty file called .nojekyll in the root — no content needed, just the file existing tells GitHub to serve the assets correctly.
3 Push to GitHub

Open a terminal, navigate into your s4labour-payroll/ folder, then run: 
git init
git add .
git commit -m "Initial commit — bureau/org prototype"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/s4labour-payroll.git
git push -u origin main
Replace YOUR_USERNAME with your GitHub username.
4 Enable GitHub Pages

On your repo on GitHub:
Settings → Pages (left sidebar) → under Source select GitHub Actions → Save

The deploy.yml workflow will automatically run on every push to main.
5 You're live

After ~1 minute your site is at:
https://YOUR_USERNAME.github.io/s4labour-payroll/

Check progress under the Actions tab in your GitHub repo.

How data persistence works

All data lives in the browser's localStorage under the key s4payroll_v1.

  • Survives page refreshes and closing/reopening the browser
  • Both bureau and org views read and write the same key — changes in one appear in the other
  • When you click an org in the bureau view, it saves the org ID to localStorage then navigates to org/index.html?org=ORG001
  • The landing page has a ↺ Reset data button to wipe back to defaults — useful before demos
  • Data is per-browser only — different browsers/devices don't share it (this is the tradeoff vs a real DB)

Future: upgrading to a real database

The Store API in shared/data.js is designed to be swapped out without touching bureau or org HTML. When ready:

  1. Replace Store.load() with a call to your Supabase/Firebase API
  2. Replace Store.save() with a write call
  3. Keep the same function names — nothing else changes
